PRODUCT DOCUMENT · MATRA MEDIA

Matra Assessment Platform

Secure online examinations with verifiable results. Live at exams.matramedia.co.in · Document revised June 2026

1. Overview

Matra is a multi-tenant platform for conducting timed, proctored online examinations at scale. It is operated by Matra Media and currently powers BharatVidya's examinations in Indian Knowledge Systems; the architecture supports additional partner institutions, each isolated from the others. The platform handles the full examination lifecycle: authoring a reusable question bank, composing papers, scheduling, identity-verified student entry, live exam-day monitoring, server-authoritative scoring, psychometric analysis, and publicly verifiable certificates.

It is designed for real Indian exam conditions — large simultaneous cohorts, mid-range Android devices, and unreliable connectivity — and has been load-validated for 2,000+ concurrent students.

2. Capabilities

Examination runtime

  • Server-synced countdown timer, immune to client clock tampering; auto-submit at deadline.
  • Per-student deterministic shuffling of question order and answer options.
  • Continuous answer saving with an offline queue that survives connection drops and resumes on reconnect.
  • Single active session per student, enforced by rotating session tokens.
  • Integrity telemetry: tab-switch logging, fullscreen-exit logging, copy-attempt blocking, per-question timing.
  • Configurable retakes, pass thresholds, and result visibility per exam.

Question bank & paper composition

  • Reusable question bank with topic, sub-topic, difficulty, Bloom's level, language, and tags.
  • Author → review → approve workflow with a four-eyes rule (a question cannot be approved by its author; edits to an approved question revert it to draft).
  • Per-institution private banks plus an optional shared master bank.
  • Blueprint-based composition (e.g. "10 easy Vedanta, 5 hard Nyaya") that copies approved questions into a frozen exam paper, rotating least-recently-used items.
  • CSV upload as an alternative to bank composition.

Assessment science

  • Item analysis per exam: difficulty index, discrimination (upper-vs-lower 27%), distractor distribution, average time.
  • Anomaly detection: improbably fast finishers, shared wrong-answer patterns, elevated integrity signals.
  • Cross-exam bank-item performance that compounds as questions are reused, surfaced in a dedicated Bank Performance view.
  • Programme-level Insights dashboard: score trends, submission volumes, pass rates, and hardest-topic ranking across all of an institution's exams.

Modular exam series

  • A 100-mark assessment delivered as weighted modules across a term or year (e.g. 20-20-20-40).
  • Aggregate decides pass/fail while per-module failures remain on record; absence counts zero until a make-up sitting.
  • One series roster synced to every module; students see a live running total (hidden-result modules excluded).

Operations & credentials

  • Live exam-day mission control: per-student state (in-exam / disconnected / submitted), answer counts, integrity flags, and per-student time extensions.
  • Certificates with a verification code and QR resolving to a public verification page; revocable, with the public page reflecting revoked status.
  • Post-event reporting: a one-click Excel report pack (results, item analysis, anomalies, certificates) and a formatted printable/PDF exam report.
  • Multi-institution tenancy with role-based admin (owner, examiner, invigilator, viewer), enforced server-side and reflected in a role-aware UI. Admins are provisioned in-app (Google sign-in, or owner-assigned passwords for institutions not on Google).

3. Architecture

A single-page web application talking directly to a managed Postgres backend; there is no custom application server. All business logic that must be trusted lives in the database as security-defined functions, so the browser cannot bypass it.

FrontendReact 19 + Vite single-page app; React Router; code-split admin bundle. Hosted on Vercel, custom domain exams.matramedia.co.in.
BackendSupabase (managed Postgres). Data accessed via PostgREST using the public anonymous key; all writes and sensitive reads go through SECURITY DEFINER RPCs.

Admin authGoogle OAuth or email + password via Supabase Auth; admin privilege resolved from an admin_users table (email → role + organisation). Password accounts are provisioned by an owner-gated edge function holding the service key server-side.

5. Security provisions

  • Answer keys never reach the client. Correct answers are stripped server-side; scoring is computed by database triggers and the submit RPC, never in the browser.
  • Row-Level Security on all 16 tables. No table is directly readable or writable by the anonymous role beyond its explicitly granted columns.
  • Identity verification. On rostered exams, entry requires a matching roll number and email, enforced inside create_attempt; a mismatch reveals nothing (no name-harvesting).
  • Exam access. Exams are unlisted by default and reached via an 8-character exam code; the roster identity is the actual credential.
  • Server-authoritative timing. The exam deadline (including per-student extensions) is enforced in the database; answers cannot be saved past time via direct API calls.
  • Response integrity. A submitted answer must belong to the student's own exam paper; scores cannot be inflated by injecting foreign questions.
  • Tenant isolation. Every admin table policy and RPC is organisation-scoped — an institution's admins can see and act on only their own institution's exams, students, question bank, and audit trail.
  • Role enforcement. The admin role tier is enforced server-side, not just in the UI: viewers are read-only, invigilators monitor and grant time extensions only, and only examiners/owners modify exam content or data.
  • Session control. One active session per student via rotating tokens; concurrent windows are locked out.
  • Rate limiting. Failed code lookups and identity probes are throttled per client IP — counting only failures, so a shared-campus IP is never affected while brute-force enumeration is cut off.
  • Auditability. Destructive and sensitive admin actions are recorded in an append-only audit log.
  • Secrets. No secret keys in the client; the service key lives only in the edge runtime; admin access is OAuth- or password-gated with the admin roster as the boundary; MFA available via Supabase Auth.

Independently penetration-tested against production (June 2026): 24 adversarial attacks as an anonymous key-holder were all refused, and a fully authenticated student decrypting the live exam paper obtains no answer key.

6. Scale & reliability

  • Validated at 2,000 concurrent virtual users through the full exam lifecycle: 0 errors, p95 response latency ~80ms.
  • Start-time spike engineered away via encrypted-paper pre-fetch and a tiny key broadcast at start.
  • Offline-tolerant: answers queue locally and sync on reconnect; the timer is server-synced and monotonic.
  • A standing k6 load-test harness allows re-validation before any large event.

7. Status

In production and penetration-tested. One tenant live (BharatVidya); multi-tenant onboarding is self-serve. Deliberately out of scope at present: payments, student-facing communication (handled by partner institutions), and third-party proctoring. Optional future work: internationalised UI strings (EN/HI/MR), a distinct series-level certificate, and admin MFA enforcement before marquee high-stakes events.
