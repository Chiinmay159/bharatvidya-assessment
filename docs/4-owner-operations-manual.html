1. The whole system in one picture

  • Frontend: React/Vite SPA, repo Chiinmay159/bharatvidya-assessment, auto-deployed by Vercel on push to main.
  • Domain: exams.matramedia.co.in (CNAME → Vercel). Old bharatvidya-assessment.vercel.app 307-redirects here.
  • Database: Supabase project Bharatvidya — ref msbpnpjjigheoplfnuly (ap-northeast-1). This is production.
  • Staging: Supabase project bharatvidya-staging — ref vtbxmnbsddkcxzqixlrp (ap-south-1). Kept paused; used for load tests and risky migrations.
  • Auth: Google OAuth via Supabase. Admin privilege is data, not config — a row in admin_users.
  • Monitoring: Sentry (set VITE_SENTRY_DSN to enable). CI: GitHub Actions runs lint + typecheck + tests + build on every push.
There is no application server to manage. All trusted logic lives in Postgres functions. To change behaviour you change either the React app (deploy via git) or the database (a migration).

3. Deploying a change

git add -A
git commit -m "what changed"
git push origin main      # CI runs, then Vercel auto-deploys to the live domain

Watch the GitHub Actions run go green and the Vercel deployment reach READY. To verify live: load exams.matramedia.co.in, plus /exam, /check, /verify, and /admin.

GitHub token: pushing the CI workflow file (.github/workflows/ci.yml) needs a token with the Workflows permission. Ordinary code pushes need only Contents: read/write.

4. Database migrations

Every schema change is a numbered SQL file in supabase/migrations/. They are the authoritative history. Current head: 023.

Procedure: apply to staging first, verify with synthetic data, then apply the identical SQL to production (Supabase dashboard → SQL editor, or the migration tooling). Migrations are written to be idempotent where possible (IF NOT EXISTS, CREATE OR REPLACE).

5. Runbooks

Onboard a new partner college

  1. Team tab → Organisations → add the college (name; optionally display name/logo via the DB).
  2. Add their admin(s) by email with role examiner (or owner for their own org), scoped to their organisation.
  3. They sign in with Google and see only their own exams, students, bank, and audit. Verified isolation is enforced at the database level.
  4. Decide whether any of your BharatVidya bank questions should be shared (set their organization_id to NULL) — otherwise each org's bank is private.

Run a large (1,000–2,000+) exam

  1. Compose/upload the paper; upload the roster; confirm counts on the batch summary.
  2. Share the exam code. Tell students to run /check the day before.
  3. Optional but recommended for a first big event: run the load test against staging (next runbook).
  4. At start, Start Now and watch Mission Control. Students pre-fetch the encrypted paper during the waiting room, so the start is smooth.
  5. Grant time extensions individually as needed; disconnected students just reopen the link.

Load-test before a big event

  1. Restore the staging project (Supabase dashboard). Note: free tier allows 2 active projects — you may need to pause another first.
  2. Ensure staging schema is current (apply any missing migrations).
  3. Create an active test batch + questions; grab its id.
  4. Run k6 run -e SUPABASE_URL=... -e SUPABASE_ANON_KEY=... -e BATCH_ID=... -e VUS=2000 loadtest/k6-exam-flow.js.
  5. Pass criteria are enforced by k6: <1% errors, p95 < 2s. Pause staging again afterward.

Pause / restore production

Free-tier projects pause after inactivity. Restore from the Supabase dashboard (or the management API). After a long pause, sanity-check that the new RPCs respond before any real exam.

Rotate the anon key

If ever needed: rotate in Supabase → Settings → API, then update VITE_SUPABASE_ANON_KEY in Vercel and redeploy. The anon key is not a secret (RLS guards everything) but rotating is harmless.

6. Security model — what protects what

  • Students never see answer keys (stripped server-side; scoring in DB).
  • RLS on all tables; anon role has only explicit column grants.
  • Entry = exam code (routing) + roll + email match against roster (the credential), enforced inside create_attempt.
  • Deadline and response-belongs-to-batch enforced server-side.
  • Tenant isolation: every admin policy and RPC is org-scoped via batch_in_my_org() / attempt_in_my_org() / series_in_my_org(). Global admins (org = NULL) see all.
  • Single session per student via rotating tokens; audit log for destructive actions.

8. Troubleshooting

SymptomLikely cause / fix
"Setup incomplete" screenEnv vars missing/wrong in Vercel. Check VITE_SUPABASE_*.
Admin login bouncesRedirect URL not in Supabase → Auth → URL Configuration. Add the live domain.
Student "details don't match"Roll/email not exactly as in roster, or no roster uploaded. Check the batch roster.
Exam not found by codeBatch not scheduled/active, or an old batch created before codes existed (active batches keep their original code). Recreate if needed.
Deploy stuck "Initializing"Vercel queue hiccup — push an empty commit or Redeploy.
Compose says "bank has only N…"Not enough approved questions for that topic/difficulty in this org's bank (+shared). Approve more or relax the blueprint.
